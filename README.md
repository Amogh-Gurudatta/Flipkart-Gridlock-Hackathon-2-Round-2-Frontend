# BTP Command Center — Frontend

> *Tactical operations terminal for the Bengaluru Traffic Police Predictive Traffic Management System.*
> Built for **Flipkart Grid 2.0 — Round 2**.

The BTP Command Center is the operator-facing dashboard that surfaces the predictions and deployment recommendations generated by the [backend AI engine](../Backend/README.md). It provides a real-time incident map, an AI forecast form, and a live dispatch log — all wrapped in a dark, monospace tactical UI.

---

## Features

### 1. Live Traffic Map (`/map`)
- Interactive Leaflet map centered on Bengaluru (`12.9716° N, 77.5946° E`) using CartoDB dark tiles.
- Severity-coded `CircleMarker` nodes for all active incidents, with smooth `flyTo` animation on selection.
- **Click-to-locate**: clicking anywhere on the map writes those coordinates into the global context so the Forecast Engine form auto-fills the location.
- After a forecast is submitted, the map overlays:
  - **Spatial impact zone** — red dashed GeoJSON polygon from the backend.
  - **Diversion route** — green GeoJSON `LineString` from live OSRM routing.
- Floating incident feed sidebar showing event ID, severity, location, and estimated duration.

### 2. Forecast Engine (`/generator`)
- Two-panel split layout: **Event Input Form** on the left, **Live Document Preview** on the right.
- The form accepts all fields required by the backend API: cause, type, priority, traffic corridor, road-closure flag, responding police station, and a GPS coordinate.
- On submission, calls `POST /api/v1/forecast` and streams the structured response into the live preview.
- The preview renders a high-fidelity **BTP Deployment Order** document (incident summary, ML prediction, recommended cops / barricades / cranes, diversion route).
- **One-click PDF export** — renders the preview with `html2canvas` at 2× resolution and saves it as a centred A4 PDF via `jsPDF`.
- Includes a subtle Web Audio API confirmation beep on successful export.
- Real-time `sonner` toast notifications for forecast success, errors, and export confirmation.

### 3. Dispatch Logs (`/warrants`)
- Searchable, filterable, animated table of every forecast submitted during the session.
- Filter by severity (`ALL` / `LOW` / `MODERATE` / `HIGH` / `CRITICAL`) or full-text search by event ID, location, or cause.
- Three sort modes: by severity, by estimated duration ascending/descending.
- Summary stat cards: total events, critical alert count, total officers required, total barricades.
- Table rows animate in/out with `framer-motion` `AnimatePresence`.

## Setup

### Prerequisites

- **Node.js** `v18+`
- The [backend server](../Backend/README.md) running on `http://localhost:8000`

### Install & Run

```bash
# From this directory
npm install
npm run dev
```

The app starts at `http://localhost:3000` and hot-reloads on save.

### Environment Variables

Create a `.env.local` file in this directory:

```env
NEXT_PUBLIC_API_URL=http://localhost:8000
```

If `NEXT_PUBLIC_API_URL` is not set, the app falls back to `http://localhost:8000`.

> **Note:** `.env*` files are gitignored. Do not commit secrets.

### Other Scripts

```bash
npm run build   # Production build
npm run start   # Serve the production build
npm run lint    # Run ESLint
```

## Backend Integration

All API calls go through the `NEXT_PUBLIC_API_URL` environment variable.

### `POST /api/v1/forecast`

The form on the **Forecast Engine** page calls this endpoint. The TypeScript types in [`src/types/forecast.ts`](src/types/forecast.ts) are kept in sync with the backend's Pydantic models and accept the same camelCase request schema.

```typescript
// EventRequest (sent by the form)
{
  eventCause: string;      // e.g. "accident"
  eventType: string;       // "unplanned" | "planned"
  priority: string;        // "Low" | "Medium" | "High"
  corridor: string;        // e.g. "Tumkur Road"
  requiresRoadClosure: boolean;
  policeStation: string;
  location: { lat: number; lng: number; address: string };
}

// ForecastResponse (received and rendered by the preview)
{
  event_id: string;
  cause: string;
  location: { lat: number; lng: number; address: string };
  predictions: {
    estimated_duration_mins: number;
    severity_level: "LOW" | "MODERATE" | "HIGH" | "CRITICAL";
  };
  deployment_recommendation: {
    traffic_cops_needed: number;
    barricades: number;
    cranes: number;
    diversion_route: string;
    diversion_geometry: GeoJsonLineString | null;
    needs_backup: boolean;
    responding_station: string;
  };
  spatial_impact_geojson: GeoJsonPolygon | null;
}
```

Once a response is received it is stored in `DataContext` and pushed to all consumers simultaneously — the map overlays the impact zone and diversion route, and the dispatch log table gains a new row, without any prop drilling.

## Global State (`DataContext`)

`DataContext` is the single source of truth for the session:

| Field | Type | Purpose |
|---|---|---|
| `incidents` | `IncidentData[]` | All incidents (seed mock data + live forecasts) shown on the map and dispatch table |
| `lastForecast` | `ForecastResponse \| null` | Most recent API response; drives the map GeoJSON overlays |
| `draftLocation` | `{ lat, lng } \| null` | Coordinate set by clicking the map; auto-fills the form |

## Design

The UI uses a dark tactical aesthetic:
- **Color palette**: deep navy base (`--bg-base`) with surface layers, a blue accent (`--accent-primary`), and severity-coded greens / ambers / oranges / reds.
- **Typography**: Inter (Google Fonts) with monospace `font-mono` for all labels, identifiers, and data.
- **Animations**: Framer Motion for table row enter/exit, hover micro-interactions via inline `onMouseEnter`/`onMouseLeave` on all interactive controls.
- Leaflet is loaded client-side only (`ssr: false`) to prevent server-side hydration errors with the DOM-dependent library.
